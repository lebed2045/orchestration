# /wf15 - Fast-Iteration TDD + Optional MCP Reviewers

**Version 15**: wf14 (quick-test inner loop, 3 cops) + optional Codex/Gemini reviewers via flags.

**Tag:** `[3-cops | 2-gate | 0-1 H | fast-iteration | optional MCP]`

**Usage:**
- `/wf15 <task>` — Autonomous, cops only (identical to wf14)
- `/wf15 -c <task>` — Cops + Codex reviewer
- `/wf15 -g <task>` — Cops + Gemini reviewer
- `/wf15 -cg <task>` — Cops + Codex + Gemini
- `/wf15 -h <task>` — Human gate after TDD_RED
- `/wf15 -cgh <task>` — All flags (human gate + Codex + Gemini)

**Key difference from wf14**: `-c` and `-g` flags inject MCP reviewers **alongside** the 3 cops at both Gate 1/2 (plan review) and Gate 2/2 (code review). Without flags, identical to wf14.

## Flag Detection

```bash
HUMAN_GATE=false
CODEX_REVIEW=false
GEMINI_REVIEW=false
TASK="$ARGUMENTS"

# Extract flags (order-independent, handles -cg, -gc, -cgh, -hgc, etc.)
if [[ "$ARGUMENTS" == *"-h"* ]]; then HUMAN_GATE=true; fi
if [[ "$ARGUMENTS" == *"-c"* ]]; then CODEX_REVIEW=true; fi
if [[ "$ARGUMENTS" == *"-g"* ]]; then GEMINI_REVIEW=true; fi

# Strip all flag tokens from task
TASK=$(echo "$TASK" | sed 's/-[cghCGH]*//g' | sed 's/^[[:space:]]*//')

echo "HUMAN_GATE: $HUMAN_GATE"
echo "CODEX_REVIEW: $CODEX_REVIEW"
echo "GEMINI_REVIEW: $GEMINI_REVIEW"
echo "TASK: $TASK"
```

## Phase Flow (9 Phases, 2 Gates, 0-1 Human Gates)

### Phase 1: CONTEXT_CAPTURE

**No test execution.** Committed code is assumed passing. Capture context only.

1. Record git SHA and branch
2. Count source files
3. Auto-detect test runner + filter syntax
4. Output CONTEXT_BLOCK (including flag state)

```bash
echo "Git SHA: $(git rev-parse HEAD)"
echo "Branch: $(git branch --show-current)"
echo "Dirty tree: $(git status --porcelain | head -1 | wc -l | tr -d ' ')"

# Auto-detect test runner + filter syntax
TEST_CMD=""
TEST_FILTER=""

# Check project CLAUDE.md for test commands
if [ -f "CLAUDE.md" ]; then
  TEST_CMD=$(grep -E '`(dotnet test|npm test|cargo test|pytest|go test)[^`]*`' CLAUDE.md | head -1 | sed 's/.*`\([^`]*\)`.*/\1/' || true)
fi

# Fallback detection
if [ -z "$TEST_CMD" ]; then
  if [ -f "package.json" ]; then
    TEST_CMD="npm test"
    TEST_FILTER="npm test -- --testPathPattern=FILE --testNamePattern=NAME"
  elif ls *.csproj 1>/dev/null 2>&1 || [ -d "Assets/Tests" ]; then
    TEST_CMD="dotnet test"
    TEST_FILTER="dotnet test --filter FullyQualifiedName~NAME"
  elif [ -f "Cargo.toml" ]; then
    TEST_CMD="cargo test"
    TEST_FILTER="cargo test NAME"
  elif [ -f "pytest.ini" ] || [ -f "pyproject.toml" ]; then
    TEST_CMD="pytest"
    TEST_FILTER="pytest FILE::CLASS::NAME -v"
  elif [ -f "go.mod" ]; then
    TEST_CMD="go test ./..."
    TEST_FILTER="go test -run NAME ./PKG"
  fi
fi

echo "TEST_CMD=$TEST_CMD"
echo "TEST_FILTER=$TEST_FILTER"
```

Save `TEST_CMD` and `TEST_FILTER` for later phases.

**Do NOT proceed without CONTEXT_BLOCK.**

### Phase 3: PLANNING

Create architecture with **minimalism in mind**.

Write to `.claude/temp/architecture.md`:
- Component design (minimal files)
- **Reuse existing patterns** (list what exists)
- File structure (**justify each new file**)
- TDD test plan

**REQUIRED — Quick Test section:**

```markdown
## Quick Test
- Type: [unit | play-mode | integration]
- Test file(s): [path(s) — one file or a small set of related tests]
- Test filter: [name, pattern, class, or scene name]
- Run command: [exact command to run only the quick test]
- Rationale: [why this is the minimal test surface for the feature]
```

The planner MUST identify the quick test — the minimal test surface the coder will iterate on. This could be one test, a few related tests in one class, or a single play-mode scene.

**Minimalist gate**: If proposing >3 new files, justify each.

### Phase 5: TDD_RED — Write Failing Quick Test

Spawn isolated coder:

```bash

Read: .claude/temp/spec.md, .claude/temp/architecture.md

Your job:
1. Read the Quick Test section from architecture.md
2. Write the quick test — could be one test method, a small test class, or a play-mode scene
3. Run ONLY the quick test using the filter/run command from architecture.md

Test runner info:
  Full suite command: $TEST_CMD
  Filter syntax: $TEST_FILTER

After writing the test, run ONLY the quick test.
Substitute the actual file path and test name into the filter syntax.

The test MUST FAIL (EXIT_CODE != 0).

Output:
1. QUICK_TEST_BLOCK (type, file(s), filter, exact run command you used)
2. EXECUTION_BLOCK showing the quick test ran and EXIT_CODE != 0

FORBIDDEN: Running '$TEST_CMD' (the full suite). Only the quick test." \
  --allowedTools Read,Write,Edit,Bash,Glob,Grep \
  --print
```

**Extract QUICK_TEST_BLOCK** from coder output. Store the exact `Run command` as `QUICK_TEST_CMD` for Phase 6.

## When to Use wf15

- Slow test suites (Unity, .NET) where you want external review perspective
- When 3-cops alone feel insufficient for complex architectural changes
- Want Codex diff-focused review alongside cops' checklist reviews
- Want Gemini structured spec-compliance check + security audit
- Need flexibility: some tasks cops-only, some tasks cops + MCP

## When to Use wf14 Instead

- No MCP tools configured
- Fast iteration without external API latency
- Simple changes where 3 cops are sufficient
- Offline/air-gapped environments
