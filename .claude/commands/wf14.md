# /wf14 - Fast-Iteration TDD Workflow (Slow Test Suites)

**Version 14**: Quick-test inner loop for projects where full test suite is slow (Unity, large .NET).

**Tag:** `[3-cops | 2-gate | 0-1 H | fast-iteration]` — Full suite runs ONCE at the end.

**Usage:**
- `/wf14 <task>` — Fully autonomous
- `/wf14 -h <task>` — Human gate after TDD_RED

**Key difference from wf12:** Coder picks the minimal test surface for the feature — one unit test, a few related tests, or a single play-mode scene — and iterates only on that during RED/GREEN. 99% of the time, only the feature's own tests break during development. Full suite runs once at the end to catch the rare regression.

## Flag Detection

```bash
HUMAN_GATE=false

if [[ "$ARGUMENTS" == *"-h"* ]]; then
  HUMAN_GATE=true
```


### QUICK_TEST_BLOCK (produced during TDD_RED)

The "quick test" is the minimal test surface for this feature — could be one unit test,
a few related tests in one class, or a single play-mode scene. The coder decides what's
relevant. The point: iterate fast on what matters, not the whole suite.

```text
┌─────────────────────────────────────────────┐
│ QUICK_TEST_BLOCK                            │
├─────────────────────────────────────────────┤
│ Type: [unit | play-mode | integration]      │

│ Run command: [exact command for quick test]  │
│ TIMESTAMP: [YYYY-MM-DD HH:MM:SS]            │
└─────────────────────────────────────────────┘
```


## Phase Flow (9 Phases, 2 Gates, 0-1 Human Gates)

### Phase 1: CONTEXT_CAPTURE

**No test execution.** Committed code is assumed passing. Capture context only.

1. Record git SHA and branch
2. Count source files
3. Auto-detect test runner + filter syntax
4. Output CONTEXT_BLOCK

```bash
echo "Git SHA: $(git rev-parse HEAD)"
echo "Branch: $(git branch --show-current)"
echo "Dirty tree: $(git status --porcelain | head -1 | wc -l | tr -d ' ')"

# Auto-detect test runner + filter syntax
TEST_CMD=""
TEST_FILTER=""

# Check project CLAUDE.md for test commands
if [ -f "CLAUDE.md" ]; then
  TEST_CMD=$(grep -E '`(dotnet test|npm test|cargo test|pytest|go test)[^`]*`' CLAUDE.md | head -1 | sed 's/.*`\([^`]*\)`.*/\1/' || true)
fi

# Fallback detection
if [ -z "$TEST_CMD" ]; then
  if [ -f "package.json" ]; then
    TEST_CMD="npm test"
    TEST_FILTER="npm test -- --testPathPattern=FILE --testNamePattern=NAME"
  elif ls *.csproj 1>/dev/null 2>&1 || [ -d "Assets/Tests" ]; then
    TEST_CMD="dotnet test"
    TEST_FILTER="dotnet test --filter FullyQualifiedName~NAME"
  elif [ -f "Cargo.toml" ]; then
    TEST_CMD="cargo test"
    TEST_FILTER="cargo test NAME"
  elif [ -f "pytest.ini" ] || [ -f "pyproject.toml" ]; then
    TEST_CMD="pytest"
    TEST_FILTER="pytest FILE::CLASS::NAME -v"
  elif [ -f "go.mod" ]; then
    TEST_CMD="go test ./..."
    TEST_FILTER="go test -run NAME ./PKG"
  fi
fi

echo "TEST_CMD=$TEST_CMD"
echo "TEST_FILTER=$TEST_FILTER"
```

Save `TEST_CMD` and `TEST_FILTER` for later phases.

**Do NOT proceed without CONTEXT_BLOCK.**

### Phase 2: INTAKE (Autonomous)

Infer requirements from codebase and user task. NO questions asked.

1. Read the user's task/request
2. Explore relevant codebase files
3. **Search for existing patterns** (coherence-cop requirement)
4. Write spec to `.claude/temp/spec.md`
5. VERIFY: Show `cat .claude/temp/spec.md | head -30`

---

### Phase 3: PLANNING

Create architecture with **minimalism in mind**.

Write to `.claude/temp/architecture.md`:
- Component design (minimal files)
- **Reuse existing patterns** (list what exists)
- File structure (**justify each new file**)
- TDD test plan

**REQUIRED — Quick Test section:**

```markdown
## Quick Test
- Type: [unit | play-mode | integration]
- Test file(s): [path(s) — one file or a small set of related tests]
- Test filter: [name, pattern, class, or scene name]
- Run command: [exact command to run only the quick test]
- Rationale: [why this is the minimal test surface for the feature]
```

The planner MUST identify the quick test — the minimal test surface the coder will iterate on. This could be one test, a few related tests in one class, or a single play-mode scene.

**Minimalist gate**: If proposing >3 new files, justify each.

---

### Phase 4: GATE 1/2 - PLAN_REVIEW (Self-Review)

Review your own plan against the 3 cops' criteria:

```text
PLAN_SELF_REVIEW
----------------
Simplicity:    [OK|ISSUE] - Complexity + file count justified?
Coherence:     [OK|ISSUE] - Existing patterns reused? Layer violations?
Coverage:      [OK|ISSUE] - Test plan covers all requirements + edge cases?
Quick Test:    [OK|ISSUE] - Minimal test surface identified? Run command valid?
----------------
VERDICT: [PROCEED|REVISE]
```

If REVISE: Fix and re-review (max 3).

---

### Phase 5: TDD_RED — Write Failing Quick Test

Spawn isolated coder:

```bash
claude -p "You are a TDD test writer with ZERO prior context.

Read: .claude/temp/spec.md, .claude/temp/architecture.md

CRITICAL CONSTRAINT: Run ONLY the quick test for this feature. Do NOT run the full test suite.

Your job:
1. Read the Quick Test section from architecture.md
2. Write the quick test — could be one test method, a small test class, or a play-mode scene
3. Run ONLY the quick test using the filter/run command from architecture.md

Test runner info:
  Full suite command: $TEST_CMD
  Filter syntax: $TEST_FILTER

After writing the test, run ONLY the quick test.
Substitute the actual file path and test name into the filter syntax.

The test MUST FAIL (EXIT_CODE != 0).

Output:
1. QUICK_TEST_BLOCK (type, file(s), filter, exact run command you used)
2. EXECUTION_BLOCK showing the quick test ran and EXIT_CODE != 0

FORBIDDEN: Running '$TEST_CMD' (the full suite). Only the quick test." \
  --allowedTools Read,Write,Edit,Bash,Glob,Grep \
  --print
```

**Extract QUICK_TEST_BLOCK** from coder output. Store the exact `Run command` as `QUICK_TEST_CMD` for Phase 6.

**Auto-stage tests:**
```bash
git add tests/ Assets/Tests/ 2>/dev/null || true
echo "Tests staged after TDD_RED"
```

### Phase 6: TDD_GREEN — Implement (Quick Test Inner Loop)

Spawn isolated coder:

```bash
claude -p "You are a TDD implementer with ZERO prior context.

Read: .claude/temp/spec.md, .claude/temp/architecture.md, test files

CRITICAL CONSTRAINTS:
1. Test files are READ-ONLY. Only edit src/ files (or Assets/Scripts/ for Unity).
2. Run ONLY the quick test. Do NOT run the full test suite.
3. Prefer editing existing files over creating new ones.
4. Reuse existing utilities/patterns.

Your quick test command: $QUICK_TEST_CMD
(This was captured during TDD_RED. Use EXACTLY this command.)

LOOP (max 5 iterations):
1. Run the quick test: $QUICK_TEST_CMD
2. If it fails, parse the error
3. Fix source code (NOT test code)
4. Re-run the quick test
5. Repeat until PASS

Output per iteration:
  AUTO-FIX ITERATION [N/5]
  Error: [parsed error]
  Fix: [what you changed]
  File: [path:line]
  Result: [PASS|STILL_FAILING]

When the quick test passes, output:
1. EXECUTION_BLOCK showing EXIT_CODE=0 for the quick test
2. Do NOT compute REGRESSION_DELTA yet (full suite has not run)

FORBIDDEN: Running '$TEST_CMD' (the full suite). Only '$QUICK_TEST_CMD'." \
  --allowedTools Read,Write,Edit,Bash,Glob,Grep \
  --print
```

**VERIFY**: Coder output shows EXECUTION_BLOCK with EXIT_CODE=0 for the quick test command.

---

### Phase 7: FINAL_VERIFICATION (Only Full Suite Run)

This is the SINGLE point where the entire test suite executes.

```bash
echo "=== FINAL_VERIFICATION: Running full test suite ==="
$TEST_CMD 2>&1 | tee /tmp/final-verification.txt
echo "EXIT_CODE: $?"
```

**FINAL_VERIFICATION_BLOCK:**

```text
┌─────────────────────────────────────────────┐
│ FINAL_VERIFICATION_BLOCK                    │
├─────────────────────────────────────────────┤
│ $ [full test command]                       │
│ Tests passing: [N]                          │
│ Tests failing: [N]                          │
│ EXIT_CODE: [0 or N]                         │
│ TIMESTAMP: [YYYY-MM-DD HH:MM:SS]            │
└─────────────────────────────────────────────┘
```

**If EXIT_CODE = 0**: Proceed to Gate 2/2.

**If EXIT_CODE != 0**: Enter FULL_SUITE_FIX (Phase 7b).

---

### Phase 7b: FULL_SUITE_FIX (Conditional — only if Phase 7 fails)

The quick test passes but the full suite reveals regressions. Fix them.

```bash
claude -p "You are a regression fixer with ZERO prior context.

The quick test passes, but the FULL test suite has failures.

Full suite output (last 50 lines):
$(tail -50 /tmp/final-verification.txt)

CRITICAL:
1. Do NOT modify the new test (it passed and is correct)
2. Fix ONLY the source code that broke existing tests
3. Run the full suite after each fix
4. Max 3 iterations

Full suite command: $TEST_CMD

After each fix, run:
  $TEST_CMD 2>&1 | tee /tmp/final-verification.txt
  echo EXIT_CODE: \$?

Output EXECUTION_BLOCK showing full suite EXIT_CODE=0." \
  --allowedTools Read,Write,Edit,Bash,Glob,Grep \
  --print
```

**Circuit breaker**: If 3 iterations fail, STOP with `MANUAL INTERVENTION REQUIRED`.

---

### Phase 8: GATE 2/2 - CODE_REVIEW (3 Cops in PARALLEL)

**Spawn all 3 cops simultaneously:**

```bash
# Cop 1: Simplicity (complexity + file bloat)
claude -p "$(cat .claude/agents/simplicity-cop.md)

Review the code changes in this PR.
Read: src/ files, .claude/temp/spec.md

Output your SIMPLICITY_COP VERDICT." \
  --allowedTools Read,Glob,Grep,Bash \
  --print > /tmp/cop-simplicity.txt &

# Cop 2: Coherence (pattern reuse + architecture)
claude -p "$(cat .claude/agents/coherence-cop.md)

Review the code changes in this PR.
Read: src/ files, search for existing patterns, check imports.

Output your COHERENCE_COP VERDICT." \
  --allowedTools Read,Glob,Grep,Bash \
  --print > /tmp/cop-coherence.txt &

# Cop 3: Coverage (tests + regression)
claude -p "$(cat .claude/agents/coverage-cop.md)

Review the code changes in this PR.
Read: src/ files, tests/ files, check coverage.
Note: Full suite already passed in FINAL_VERIFICATION.

Output your COVERAGE_COP VERDICT." \
  --allowedTools Read,Glob,Grep,Bash \
  --print > /tmp/cop-coverage.txt &

wait
cat /tmp/cop-simplicity.txt /tmp/cop-coherence.txt /tmp/cop-coverage.txt
```

#### Aggregate Verdicts

```text
3-COPS GATE SUMMARY
-------------------
Simplicity:    [PASS|REJECT]

Coverage:      [PASS|REJECT]
-------------------

```

**If ANY cop returns REJECT**: Fix issues and re-run that cop (max 3 per cop).

**No SMOKE_TEST** — FINAL_VERIFICATION already ran the full suite.

## Circuit Breakers

| Trigger | Action |
|---------|--------|
| No CONTEXT_BLOCK | STOP |
| Quick test fails 5x in GREEN | STOP, escalate |
| Full suite fails + 3 fix attempts | STOP: MANUAL INTERVENTION |
| Same cop fails 3x | STOP: REVIEW LOOP EXCEEDED |
| Tests fail 5x total | STOP: MANUAL INTERVENTION |
| "Pre-existing failures" claim | RETRACT. Fix the code. |
| File count +5 without justification | STOP, consolidate |
| Coder runs full suite during RED/GREEN | RETRACT. Respawn with constraint. |

## Test Filter Reference

| Runner | Full Suite | Single Test Filter |
|--------|-----------|-------------------|
| jest/npm | `npm test` | `npm test -- --testPathPattern=FILE --testNamePattern=NAME` |
| dotnet | `dotnet test` | `dotnet test --filter FullyQualifiedName~NAME` |

| pytest | `pytest` | `pytest FILE::CLASS::NAME -v` |

## Comparison: wf14 vs wf12

| Aspect | wf12 | wf14 |
|--------|------|------|
| Full suite runs | 3+ (baseline, green, smoke) | **1** (FINAL_VERIFICATION only) |
| Baseline test run | Yes | **No** (CONTEXT_CAPTURE) |
| TDD iteration | Full suite | **Quick test only** |
| Test command | Hardcoded | **Auto-detected + filter** |
| New proof block | None | **QUICK_TEST_BLOCK** |
| Recovery phase | None | **FULL_SUITE_FIX** |
| Coder constraint | "Run tests" | **"Run ONLY the quick test"** |
| Best for | Fast suites | **Slow suites (Unity, .NET)** |
| Cops | 3 (same) | 3 (same) |
| Human gate | Optional -h (same) | Optional -h (same) |

---

## When to Use wf14

- Unity projects (5+ minute test suites)
- Large .NET solutions with slow test discovery
- Any project where full test suite takes >60 seconds
- When you want tight RED-GREEN iteration on a single feature
- When baseline is known-good (committed, CI green)

## When NOT to Use wf14

- Fast test suites (<10 seconds) — use wf12 instead
- When you don't trust the baseline (uncommitted changes, flaky tests)
- When the feature is so cross-cutting that no small test surface can represent it
